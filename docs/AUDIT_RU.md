# I2PChat — аудит безопасности (технический)

**Версия документа:** 1.2  
**Дата:** 2026-04-02  
**Уточнение объёма:** Согласовано с **v1.1.3** — живой поток **только vNext** (устаревший построчный wire-формат удалён); дисковый ввод при исходящей отправке и пакетирование **drain** для ACK без изменений; остальной объём как прежде. **Срез на момент времени**, не pentest и не сертификация.

**Объём:** Исходный код в `i2pchat/`, точки входа GUI, хранилище, модуль протокола и BlindBox, выборочные тесты.

**Методика:** Статический разбор архитектуры, использования криптографии, границ сети, работы с файлами, секретами и типовых классов уязвимостей (инъекции, обход путей, downgrade, десериализация, выполнение команд). Динамическая эксплуатация не выполнялась.

---

## 1. Кратко о продукте

I2PChat — настольный чат-клиент с использованием **I2P SAM** для потоковых соединений с пирами. Реализован **бинарный кадровый протокол vNext**, **рукопожатие на NaCl** с эфемерными ключами, **аутентифицированное шифрование** полезной нагрузки, модель доверия **TOFU** к ключу подписи пира, опциональная **шифрованная локальная история** и **BlindBox** — офлайн-доставка через сторонние реплики.

Стек: Python 3, GUI **PyQt6**, **PyNaCl**, внутренний **`i2pchat.sam`** (SAM), **Pillow** для изображений.

---

## 2. Модель угроз и допущения

**Угрозы в области внимания**

- Злонамеренный или дефектный пир по сети (злоупотребление протоколом, огромные кадры, replay, попытки downgrade).
- Злонамеренная реплика BlindBox или наблюдатель на путях I2P к репликам (в пределах заявленного протокола).
- Локальный атакующий с правами пользователя (чтение профиля, правка файлов, буфер обмена).
- Компрометация цепочки поставки зависимостей или сборки.

**Вне области / явные ограничения**

- Компрометация роутера I2P или ядра ОС, физическая кража разблокированной машины без шифрования диска.
- Атаки на удобство (спам, социальная инженерия), кроме случаев, когда клиент их усиливает.
- Формальная верификация криптопротоколов.

**Опоры доверия**

- Пользователь доверяет **роутеру I2P** и настроенной конечной точке **SAM**.
- При офлайн-доставке пользователь доверяет **операторам реплик BlindBox**.
- **TOFU:** привязка идентичности сильна ровно настолько, насколько корректно пользователь подтвердил **первый** успешный контакт.

## 3. Итоговая сводка

| Область | Оценка |
|---------|--------|
| Крипта живого канала | **Сильная:** современные примитивы NaCl, отдельный ключ MAC, проверка порядковых номеров, блокировка downgrade после рукопожатия. |
| Фрейминг / лимиты DoS | **Хорошо:** `max_frame_body`, лимиты ресинхронизации; крупные передачи режутся на чанки. **Только vNext** на живом канале — отдельного парсера старых кадров нет. |
| Приём файлов | **Хорошо:** санитизация имён, уникальные пути, лимиты размера; дроссель UI снижает «подвисания» (доступность). |
| Локальная история | **Хорошо:** соль на файл, слои HKDF, SecretBox; экспорт на Argon2id. |
| BlindBox | **Зависит от развёртывания:** строковый протокол + опциональные токены реплик; это не классический TLS — доверие к репликам отдельный слой. |
| Инъекции в GUI чата | **Низкий риск для баблов:** текст рисуется как строка; rich text в основном в поле ввода с очисткой вставок по коду. |
| Секреты в репозитории | **Нет зашитых API-ключей** в просмотренных путях; переменные окружения для операторов. |
| Subprocess | **Ограниченно:** уведомления со списком argv без `shell=True` в просмотренном коде приложения. |

**Остаточные риски:** ошибки конфигурации (слабые токены BlindBox), **plaintext `.dat`**, если не используется keyring; проверка обновлений тянет HTML с URL (доверие к источнику); **UX доверия к пиру** (пользователь должен осознанно подтверждать смену TOFU).

## 4. Подробные выводы

### 4.1 I2P SAM и адресация

- Сборщики **`i2pchat.sam.protocol`** отклоняют опасные символы в destination и session id (`tests/test_sam_input_validation.py`), снижая риск инъекций в строки команд SAM.
- Строки пира по-прежнему опираются на **модель угроз I2P**; приложение её не переопределяет.

**Критичность:** справочная (дополнительная глубина защиты).

---

### 4.2 Живой протокол (vNext) и рукопожатие

- **Фрейминг:** `ProtocolCodec` проверяет `MAGIC`, версию, допустимые типы и максимальный размер тела (`i2pchat/protocol/protocol_codec.py`). Живой поток — **только vNext**; отдельного режима для устаревшего построчного формата **нет**. Поток без корректного `MAGIC` отсекается лимитом ресинхронизации или проверками протокола.

- **После рукопожатия:** незашифрованные прикладные кадры трактуются как **нарушение протокола** и ведут к разрыву (`receive_loop`).

- **Крипта:** эфемерный X25519, HKDF для `k_enc` / `k_mac`, XSalsa20-Poly1305 (SecretBox), HMAC по шифротексту/метаданным с **монотонной последовательностью** (`i2pchat/crypto.py`, `i2pchat/core/i2p_chat_core.py`).

- **Replay / порядок:** жёсткая проверка `expected_seq`; сбои — ошибка и disconnect.

**Рекомендации:** следить за логами downgrade и сбоев MAC.

---

### 4.3 Подтверждения доставки (ACK)

- Таблицы ожидающих ACK с **TTL и верхней границей** по числу записей (`ACK_TTL_SECONDS`, `ACK_MAX_PENDING`, периодическая чистка). Поддельные/дубликаты ACK фильтруются со счётчиками (`_record_ack_drop`).

**Критичность:** низкая — в основном память и чистота состояния.

---

### 4.4 BlindBox (офлайн)

- Отдельный **строчный протокол** реплик, не смешивается с vNext (`docs/PROTOCOL.md`).
- **Корневой секрет** и отложенные секреты оборачиваются для хранения ключами, завязанными на профиль.
- **Аутентификация реплик:** опциональные токены на endpoint и токен для loopback (`I2PCHAT_BLINDBOX_LOCAL_TOKEN`, `replica_auth` в JSON). Пустой токен в небезопасной среде — **операционный** риск; код предупреждает и может отказать в строгих сценариях.
- **Лимит GET:** `max_get_blob_size` связан с лимитами blob (`BlindBoxClient`).

**Риски:** реплики видят **шифротексты** и метаданные в рамках протокола; хосты реплик следует считать **полудоверенной** инфраструктурой.

---

### 4.5 Локальное хранилище и профили

- **История чата:** шифрование на диске, ключи от identity и per-peer (`i2pchat/storage/chat_history.py`). Неверный ключ — расшифровка не удаётся, история не показывается.
- **Экспорт истории:** пароль + Argon2id + SecretBox (`i2pchat/storage/history_export.py`). Стойкость пароля — на пользователе.
- **Экспорт/импорт профиля:** архивы с паролем (`i2pchat/storage/profile_export.py`).
- **Keyring:** опциональное хранение секретов, сервис `i2pchat` (`KEYRING_SERVICE`). Fallback — **чувствительный** plaintext в `.dat` — ожидаемо для переносимости; важны права ОС на каталог данных.
- **Атомарные записи:** в ряде путей снижают риск порчи файлов при сбое.

**Риск:** резервные копии и `.dat` на диске — **ценные цели**; права 0700 на Unix помогают, но не останавливают вредонос от имени пользователя.

---

### 4.6 Файлы и изображения

- **Имена:** `sanitize_filename` убирает компоненты пути и опасные символы; `allocate_unique_filename` исключает перезапись.
- **Размеры:** лимиты файла/картинки, проверка чанков к остатку, base64 с ограничениями.
- **Inline-изображения:** проверка magic, валидация PIL после приёма (в новых версиях часть работы в thread pool). Битый файл — отказ с сообщением пользователю.
- **Исходящее чтение с диска (v1.1.2):** синхронный `read()` для чанков **файла** и **inline (`G`)** выполняется в **пуле потоков** (`asyncio.to_thread`), чтобы не блокировать цикл **asyncio** на том же потоке, что Qt+qasync — **доступность** / отзывчивость.
- **ACK при исходящей передаче (v1.1.2):** автоматические **MSG_ACK** / **IMG_ACK** в **`S`**-кадрах идут через **`_write_signal_frame_maybe_soft_drain`**: пока **`_file_transfer_active`**, **`drain()`** вызывается каждые **`I2PCHAT_MSG_ACK_DRAIN_EVERY`** кадров (по умолчанию **16**, с ограничением диапазона), снижая накладные расходы. Слишком редкий drain теоретически задерживает видимость ACK для пира; значения по умолчанию осторожные.

**Риск:** лишние проходы по диску на огромных локальных файлах — скорее производительность, чем безопасность; сеть ограничена лимитами.

---

### 4.7 GUI и интеграция с ОС

- **Отрисовка чата:** баблы для обычных сообщений рисуются **QPainter** как текст — не произвольный HTML от пира; риск **XSS в ленте** ниже, чем у полноценного HTML-чата.
- **Поле ввода:** включён rich text для эмодзи; при вставке теги снимаются до отправки в протокол — снижает случайные вставки разметки; на пир уходит то, что прошло пайплайн.
- **Открытие путей:** `QDesktopServices.openUrl` + `fromLocalFile` для контролируемых каталогов; действия по инициативе пользователя.
- **Subprocess:** звук на Linux через `subprocess.Popen` со **списком argv** и путями из `shutil.which` — без интерполяции shell (`i2pchat/gui/main_qt.py`).
- **Буфер обмена:** стандартное копирование текста — зона ответственности пользователя и ОС.

---

### 4.8 Обновления и внешний HTTP

- **Проверка обновлений** загружает **HTML страницы релизов**, парсит версии, **не качает бинарники автоматически** (`docs/MANUAL_EN.md` / RU).
- **Риск:** если `I2PCHAT_RELEASES_PAGE_URL` указывает на злонамеренный origin, теоретически возможны баги парсера HTML — трактовать URL как **доверенную конфигурацию**. В просмотренных путях ядра **eval** по содержимому ответа не выявлен.

---

### 4.9 Переменные окружения и отладка

| Переменная | Влияние |
|------------|---------|
| `I2PCHAT_MSG_ACK_DRAIN_EVERY` | При **исходящей** отправке файла/картинки: **drain** после каждых **N** автоматических **MSG_ACK**/**IMG_ACK** (по умолч. **16**, диапазон **1–256**). Влияет на **отзывчивость и обратное давление**, не на крипту. |
| `I2PCHAT_BLINDBOX_*` | Списки реплик, токены, кворум — **ошибка конфигурации** ослабляет офлайн-безопасность. |
| `I2PCHAT_FILE_XFER_DEBUG` | Логи таймингов — в логах может **прослеживаться** характер передачи. |
| `I2PCHAT_QT_FILE_EVENT_NOOP` | Отключает UI прогресса файлов — только диагностика. |

В продакшене отладочные флаги лучше не включать без защиты логов.

---

### 4.10 Зависимости и сборка

- Крипта на **PyNaCl** (широко используемая библиотека).  
- **PyQt6**, **Pillow** — следить за CVE upstream.  
- **pickle** по ненадёжным сетевым данным в просмотренных путях чата не найден.  
- PyInstaller / скрипты сборки: практики **воспроизводимой сборки** и **подписи** (в README — `I2PCHAT_REQUIRE_GPG` для релизов).

## 5. Приоритетные рекомендации

1. **Высокий (эксплуатация):** Задавать **токены BlindBox** при TCP/loopback-репликах; по возможности использовать **keyring**.
2. **Средний:** Периодически `pip audit` / аудит пакетов ОС для пинов зависимостей релиза.
3. **Средний:** Публиковать **подписанные** бинарники и хеши (проект уже это описывает).
4. **Низкий:** В README — блок «Как сообщить об уязвимости» + ссылка на этот аудит.
5. **Низкий:** Для парсера страницы обновлений — минимальная логика и регрессионные тесты при расширении.

---

## 6. Заключение

I2PChat реализует **согласованный по современным ожиданиям** стек для зашифрованного чата поверх I2P: жёсткие правила **vNext**-фрейминга, AEAD и MAC, явная борьба с downgrade, шифрованная локальная история. Доминирующие риски — **развёртывание и доверие** (подтверждения TOFU, операторы BlindBox, физический доступ к файлам профиля), а не одна очевидная удалённая RCE в просмотренных путях.

Документ стоит **обновлять** после крупных изменений протокола, крипты, BlindBox или фрейминга.
