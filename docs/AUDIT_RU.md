# Аудит безопасности I2PChat — собственный слой SAM (`i2pchat.sam`)

**Контекст:** Переход с PyPI / вендорного **`i2plib`** на реализацию в репозитории в пакете **`i2pchat/sam/`** (см. `docs/SAM_INTERNAL_BACKEND_PLAN.md`, `docs/releases/RELEASE_1.2.4.md`).

**Дата аудита:** 2026-04-07 (статический разбор текущего дерева; без пентеста).

**Устранение (2026-04-07):** словарь **options** для `SESSION CREATE` валидируется по каждой паре ключ/значение (включая сырые `\r`/`\n`/`\x00` до `strip`). BlindBox **`STREAM CONNECT`** собирается через **`build_stream_connect`**. Ключи **PUT/GET** в BlindBox не допускают пробелов и переводов строк. Тесты: `tests/test_sam_input_validation.py`, `tests/test_blindbox_client.py`.

**Объём:** код управляющего плоскости SAM (`protocol.py`, `client.py`, `backend.py`, `destination.py`, `errors.py`), использование в `i2pchat/core/i2p_chat_core.py` и `i2pchat/blindbox/blindbox_client.py`, смежные границы доверия (BlindBox по TCP, встроенный роутер). Криптография уровня приложения (NaCl, HKDF, vNext) упоминается только там, где пересекается с моделью доверия к SAM.

---

## 1. Краткое резюме

Внутренний стек SAM **сохраняет подход к усилению**, который раньше применялся в `i2plib`: токены SAM-команд проверяются на запрещённые символы (переносы строк, пробелы в критичных полях и т.д.), основной чат использует билдеры **`build_*`** из `i2pchat.sam.protocol`. Отказ от внешней зависимости **уменьшает поверхность supply-chain**, но **всю корректность протокола** теперь несёт этот репозиторий — регрессии в разборе или валидации полностью на стороне проекта.

**Критических уязвимостей удалённого выполнения кода** в просмотренном коде SAM не выявлено. Остаточные риски в основном связаны с **моделью доверия SAM** (открытый TCP до роутера, обычно loopback) и с **конфигурацией** (указание чужого SAM-хоста).

---

## 2. Модель угроз (в части SAM)

| Участник | Возможности | Митигации / пробелы |
|----------|-------------|---------------------|
| Локальный процесс | Подключение к порту SAM, гонки с роутером пользователя | Изоляция пользователей ОС, фаервол, привязка SAM к loopback |
| Вредоносный / скомпрометированный роутер I2P | Подделка ответов SAM, наблюдение за идентичностями, отбрасывание трафика | Выбор роутера пользователем; TLS не входит в стандартный SAM |
| Пир в сети I2P | Атаки на протокол чата (вне SAM) | vNext-фрейминг и NaCl в `i2pchat.core` / `i2pchat.crypto` |
| Вредоносная реплика Blind Box | Плохие blob’ы, DoS | Кворум, лимиты размера, криптография поверх содержимого |

## 3. Что улучшилось после миграции

- **Цепочка поставки:** `i2plib` не объявлен в `pyproject.toml`; политика в тестах это фиксирует (`tests/test_audit_remediation.py`). Поведение SAM можно аудировать в дереве исходников.
- **Инъекции в SAM-команды:** в `i2pchat/sam/protocol.py` валидируются идентификатор сессии, стиль, порты, версии HELLO, имена NAMING, назначения STREAM, булевы флаги до форматирования строк.
- **Логи:** `_redact_sam_reply` маскирует чувствительные ключи ответов SAM (`PRIV`, `PRIVATE`, `DESTINATION`, `SIGNING_PRIVATE_KEY`).
- **CI:** `.github/workflows/security-audit.yml` запускает **`uv export` + `pip-audit`** по зафиксированным зависимостям runtime и build.

## 4. Находки

Шкала: **Критический / Высокий / Средний / Низкий / Информационный**.

### 4.1 [Средний] Словарь опций `SESSION CREATE` — ключи и значения не валидируются как отдельные токены

**Место:** `i2pchat/sam/protocol.py` — `build_session_create(..., options=dict)`.

**Суть:** В отличие от скалярных полей SAM, каждая пара **`key=value`** собирается без тех же правил `_validate_sam_token`. Итоговая строка проходит `_validate_sam_options`, которая отсекает `\r`, `\n`, `\x00`, но **не** запрещает пробелы внутри значения, лишние `=` и т.п.

**Текущий риск:** В коде дерева (`i2p_chat_core.py`, дефолтные `sam_options` у BlindBox) задаются **фиксированные** пары, то есть **в штатных сценариях** эксплуатации нет. Риск растёт, если позже в `create_session(..., options=...)` попадут **частично пользовательские** данные.

**Рекомендация:** Валидировать каждый ключ/значение теми же правилами, что и остальные SAM-токены (или документированным подмножеством), либо разрешать только белый список имён I2CP-опций.

---

### 4.2 [Средний] BlindBox: `STREAM CONNECT` собирается вручную (f-string)

**Место:** `i2pchat/blindbox/blindbox_client.py` — `_open_sam_stream_to`.

**Суть:** Строка не строится через `sam_protocol.build_stream_connect(...)`. Сейчас смягчают риск `_validate_sam_destination` и генерация `_active_sam_id`, но **два пути** для одной и той же SAM-команды повышают вероятность **расхождения** при будущих правках.

**Рекомендация:** Вызывать `build_stream_connect(session_id, destination, silent="false")`, чтобы вся проверка оставалась в одном месте.

---

### 4.3 [Низкий] `DEST GENERATE` — тип подписи без белого списка

**Место:** `i2pchat/sam/protocol.py` — `build_dest_generate`.

**Суть:** Любой `int` уходит в команду; отсечение некорректных типов — на стороне роутера.

**Рекомендация:** По желанию — allowlist (например, Ed25519 = 7 для политики по умолчанию).

---

### 4.4 [Низкий] Неограниченный `readline()` на SAM и BlindBox TCP

**Место:** `i2pchat/sam/client.py`, `open_stream_*`, вспомогательные функции BlindBox.

**Суть:** Узел, не отправляющий `\n`, может раздувать буфер читателя (DoS), если транспорт не считается доверенным.

**Рекомендация:** Для loopback SAM зафиксировать это как принятую модель; для жёстких сценариев — чтение строки с лимитом байт.

---

### 4.5 [Низкий / defense in depth] Ключи BlindBox в `PUT` / `GET`

**Место:** `i2pchat/blindbox/blindbox_client.py`.

**Суть:** Ключи подставляются в текстовые команды. В **core** используется hex-`lookup_token` из `derive_blindbox_message_keys()` (безопасный алфавит). Прямой вызов `BlindBoxClient` со строками «как попало» может сломать протокол строки.

**Рекомендация:** Отклонять ключи с пробелами, `\r`, `\n`, `\x00` и разделителями, если API должно оставаться безопасным для произвольных вызовов.

---

### 4.6 [Информационный] Ослабленный `expect_ok` под особенности i2pd

**Место:** `i2pchat/sam/protocol.py` — `expect_ok`.

**Суть:** Успех без `RESULT=OK` выводится для некоторых форм `DEST REPLY` и `SESSION STATUS` (совместимость с i2pd). Теоретически **вредоносный роутер** мог бы влиять на классификацию успех/ошибка; на практике клиент и так **полностью доверяет** роутеру в SAM.

**Рекомендация:** Оставить для совместимости; менять только при появлении аутентифицированного канала управления.

---

### 4.7 [Информационный] `Destination(..., path=...)`

**Место:** `i2pchat/sam/destination.py`.

**Суть:** Чтение файла с диска — мощная операция; путь не должен быть **произвольным пользовательским** вводом.

**Рекомендация:** При новых фичах проверять все вызовы; предпочитать ключи из keyring в виде bytes/str.

---

### 4.8 [Информационный] Подпроцесс встроенного `i2pd`

**Место:** `i2pchat/router/bundled_i2pd.py`.

**Наблюдение:** Запуск через `asyncio.create_subprocess_exec` со списком аргументов (в просмотренном пути нет `shell=True`). Целостность **бинарника** зависит от сборки и окружения пользователя.

**Рекомендация:** Продолжать фиксировать/подписывать артефакты релиза (как в скриптах сборки и CI policy tests).

## 5. Остаточные риски (не как дефекты кода)

1. **Открытый TCP для SAM** — ожидаемо для I2P SAM; безопасный деплой предполагает **loopback** или контролируемую сеть.
2. **Роутер как TCB** — любой I2P-клиент раскрывает роутеру сведения о туннелях и идентичностях.
3. **Ответственность за регрессии** — граничные случаи протокола, раньше частично на стороне `i2plib`, должны покрываться тестами **`i2pchat.sam`** (`tests/test_sam_*.py`, `tests/test_blindbox_client.py`).
