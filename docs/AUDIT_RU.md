# Аудит безопасности I2PChat (полный репозиторий)

**Дата аудита:** 2026-04-09  
**Метод:** статический code review + целевые security-тесты + аудит зависимостей (`pip-audit` по lock-экспортам `uv`).  
**Объём:** весь репозиторий с фокусом на ключевые границы доверия: обновления, SAM-транспорт, BlindBox, запуск встроенного роутера, локальное хранение данных, CI и supply-chain контроли.

---

## 1. Резюме

- Подтверждённых уязвимостей уровня **Critical/High** в текущем дереве не выявлено.
- Главные практические риски сейчас — **модель доверия и эксплуатационные настройки**:
  - проверка обновлений парсит HTTP-страницу релизов (проверка целостности выполняется пользователем, не приложением);
  - опциональные небезопасные режимы BlindBox зависят от осознанных настроек оператора;
  - граница доверия SAM по-прежнему проходит через локальный роутер.
- Сильные стороны:
  - усиленная валидация SAM + регрессионные тесты;
  - CI-сканирование секретов и lock-aware аудит зависимостей;
  - policy-проверки подписи релизных артефактов в CI.

**Итог по severity:** Critical `0`, High `0`, Medium `1`, Low `3`, Informational `5`.

## 3. Находки

Шкала: **Критический / Высокий / Средний / Низкий / Информационный**.

### 3.1 [Средний] Проверка обновлений не аутентифицирует метаданные криптографически

**Где:** `i2pchat/updates/release_index.py`, `i2pchat/gui/main_qt.py`

**Доказательство:** логика обновлений скачивает и парсит HTML (`fetch_releases_page`, `check_for_updates_sync`), затем может открыть `downloads_page_url()` в браузере.

**Влияние:** при враждебной сети/прокси/источнике страницы можно подменить «доступную версию» и подвести пользователя к вредоносной странице загрузки.  
Важный нюанс: приложение **не** делает авто-скачивание/авто-установку; UI отдельно предупреждает про проверку `SHA256SUMS`/GPG.

**Рекомендация:** оставить текущие предупреждения; если когда-либо добавится автоапдейт — только через **подписанный манифест обновлений** с встроенным доверенным публичным ключом.

---

### 3.2 [Низкий] Переопределяемый URL релизов через окружение влияет на update UX

**Где:** `i2pchat/updates/release_index.py` (`I2PCHAT_RELEASES_PAGE_URL`), `i2pchat/gui/main_qt.py` (диалог предупреждения + `openUrl`).

**Влияние:** локальный атакующий/неверная конфигурация окружения может направить проверку обновлений и кнопку загрузок на произвольный URL.

**Текущая защита:** одноразовый warning в GUI с явным пояснением рисков; в мануале есть раздел про верификацию загрузок.

**Рекомендация:** опционально усилить allowlist схем/хостов или добавить более строгий warning при уходе от дефолтного хоста.

---

### 3.3 [Низкий] Дефолтные кворумы BlindBox ориентированы на доступность

**Где:** `i2pchat/core/i2p_chat_core.py` (`I2PCHAT_BLINDBOX_PUT_QUORUM`, по умолчанию `1`).

**Влияние:** при слабом кворуме и недоверенных репликах поведение одной реплики сильнее влияет на результат доставки, чем при строгом кворуме.

**Рекомендация:** для production-сценариев документировать и применять несколько независимых реплик + более строгий quorum (если допустимо по задержке).

---

### 3.4 [Низкий] Декодирование изображений может давать локальную нагрузку на ресурсы

**Где:** `i2pchat/core/i2p_chat_core.py` (`validate_image`, Pillow `Image.open(...).load()`).

**Влияние:** при открытии специально подготовленных изображений возможен локальный DoS по CPU/памяти.

**Текущая защита:** до декодирования проверяются размер файла и максимальные dimensions.

**Рекомендация:** defense-in-depth — явная политика защиты от decompression-bomb и/или stricter decode лимиты.

## 4. Информационные наблюдения (не прямые уязвимости)

1. **Модель доверия SAM прозрачна:** приложение осознанно доверяет выбранному I2P-роутеру на контрольной границе.
2. **Небезопасный локальный режим BlindBox — только opt-in:** без токена режим блокируется, если явно не выставлен `I2PCHAT_BLINDBOX_ALLOW_INSECURE_LOCAL=1`.
3. **Гигиена запуска процессов в целом хорошая:** для встроенного роутера используется `create_subprocess_exec`/списки аргументов; широкого runtime-паттерна `shell=True` в проверенных путях не найдено.
4. **CI уже закрывает supply-chain базу:** pinned `gitleaks`, `pip-audit`, policy-проверки подписи релиза в `.github/workflows/security-audit.yml`.
5. **Исключения secret scan узкие:** `.gitleaks.toml` allowlist ограничен конкретной test-fixture.

## 6. Остаточные риски

1. **Доверие к сетевому источнику метаданных обновлений** до появления подписанного манифеста.
2. **Роутер как TCB:** злонамеренный/скомпрометированный роутер может влиять на SAM-границу.
3. **Осознанно небезопасные флаги оператора** (например, insecure local mode, слабые кворумы).
